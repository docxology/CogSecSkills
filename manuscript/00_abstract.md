# Abstract {#sec:abstract}

CogSecSkills is a defensive, harness-neutral agent-interface library for cognitive security and analytic tradecraft, distributed from `github.com/docxology/CogSecSkills`. The motivation is an information environment in which mis-, dis-, and malinformation are analytically distinct but operationally entangled, false content can diffuse rapidly at platform scale, and source evaluation increasingly requires explicit expert practice rather than page-bound inspection alone [@wardleDerakhshan2017informationDisorder; @vosoughi2018spread; @wineburg2019lateralReading]. The live generated catalogue reports 100 implemented skills across seven taxonomy groups. Each skill is owned by a canonical YAML definition, rendered into a harness-neutral specification, and adapted to a configured harness set whose default members are Claude, Codex, and Hermes. The library is organized as a Plan/Build/Teach system: `registry/skills.yaml` declares the catalogue, `definitions/` owns the skill substance and quality controls, `skills/<group>/<slug>/` exposes the harness-facing build, and `docs/ageint/` provides the AGEINT educational upstream for defensive analytic practice [@ageint2026].

The core design choice is to make the reusable skill contract smaller than any one agent interface but stricter than a prompt collection. A skill declares triggers, inputs, outputs, per-skill reference metadata, group-aware quality controls, and a closed vocabulary of neutral tool verbs before any harness-specific adapter is considered. This positions CogSecSkills as a validated interface between reasoning, tool use, and defensive output discipline, not as a claim that any particular model runtime behaves correctly in the field [@yao2022react; @schick2023toolformer]. Installing CogSecSkills into an agent harness is therefore concrete rather than interpretive: clone the public repository, install or run the Python package, run validation, point the harness at `skills/<group>/<slug>/SKILL.md`, execute through `workflow.md`, and bind runtime tools through `harness/<name>.md`. For non-default harnesses, `cogsecskills.yaml` declares the configured harness set and `python -m cogsecskills definitions --write` regenerates adapters before `validate` checks the result.

This manuscript describes the system contract, source ownership, and local verification gates; it does not claim field effectiveness. It explains the closed tool-verb vocabulary, the multiharness adapter layer, the definition-to-skill renderer, the quality and governance checks, and the generated manuscript assets that make the library visually scannable. The quality audit now checks that defensive boundaries, misuse redirects, evidence requirements, confidence rubrics, privacy/legal constraints, uncertainty handling, failure modes, and negative controls are not merely present but specific to the skill. It also rejects reused individual negative-control, confidence, evidence, and privacy/legal entries across the corpus. The evidence ladder adds curated safe-use and unsafe-redirect scenarios with expected defensive response-shape contracts, reviewed expected-answer bodies, and one source-owned worked example per skill; the quality dashboard summarizes the live 100-skill corpus, quality-capsule coverage, scenario coverage, worked-example coverage, harnesses, references, source paths, and latest verified-state rows as a generated drift surface. The supplemental catalogue and metadata matrix are produced directly from the live registry and skill specifications. Deterministic figures summarize taxonomy counts, the 100-skill atlas, tool-verb coverage, AGEINT topic crosswalks, Plan/Build/Teach flow, reference density, harness coverage, and the cover-page installation route for agent harness users.

The evidence boundary is repository-local and reproducible within the checked-out project state, following reproducible-computing, FAIR stewardship, and software-citation norms for explicit workflows, version specificity, and citable software artifacts [@sandve2013reproducible; @wilkinson2016fair; @smith2016softwareCitation]. Claims in this manuscript are supported by source files, canonical definitions, generated supplements, generated figures, the generated quality dashboard, and project-local commands: `cogsecskills definitions --check`, `cogsecskills validate`, `cogsecskills report`, `cogsecskills doctor`, `cogsecskills scenarios --check`, `cogsecskills examples --check`, `cogsecskills dashboard --check`, `cogsecskills manuscript-assets --check`, the focused pytest suite, and the sibling template renderer. The figures, supplements, scenario expected answers, worked examples, and dashboard should be read as synchronized views of the current library state, not as independent measurements of operational performance.
